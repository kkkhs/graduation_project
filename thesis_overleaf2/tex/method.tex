\section{数据集、评价指标与方法方案}\label{sec:method}

\subsection{数据集介绍}

本文采用 LEVIR-Ship 数据集作为统一实验基准\cite{chen2022_tinyship_drenet}。该数据集面向中分辨率遥感微小船舶检测任务，能够较好反映复杂海面背景下小目标检出的实际难点。数据集划分如表~\ref{tab:ch3_dataset_split} 所示。

LEVIR-Ship 的典型特点是目标尺度小、背景纹理复杂、场景差异明显。船舶在图像中占比很小，单靠一个指标很难完整描述模型表现，所以本文在实验中同时关注检测精度、召回能力和工程可部署性。

\subsection{数据集样例与任务特征}

LEVIR-Ship 覆盖的场景类型比较丰富——开阔海域的稀疏目标、近岸区域、港口区域、多目标密集分布场景都有。不同图像块在背景纹理、目标密度、目标朝向和局部对比度上差异很大。这意味着模型既要对弱目标敏感，又要能抑制复杂背景纹理的干扰。对微小船舶检测来说，"目标小而背景强"的组合恰恰是最主要的识别难点。

基于这个特点，本文在后续实验中不仅关注 AP50 的变化，还特别重视 Precision、Recall、F1 以及定性样例中的误检和漏检类型。对于遥感船舶检测这种同时有检出需求和工程部署需求的任务，只看一个精度指标往往不足以判断模型是否真的适合系统落地。

\subsection{数据预处理与统一格式}

跨框架实验最怕的就是数据口径不一致。本文的做法是：统一标注协议，统一评测口径。

具体来说做了四件事。一是核验图像与标注文件的一一对应关系，检查空标注、异常坐标和类别不一致的问题。二是把数据整理成 COCO 中间格式，让 MMDetection 和 YOLO 系列共享同一套评测流程。三是固定 train、val、test 的划分和样本清单，避免不同模型因为划分差异产生额外偏差。四是把各模型输出统一成包含 \texttt{image\_id}、\texttt{model\_name}、\texttt{bbox}、\texttt{score} 和 \texttt{category\_id} 的结构化结果，方便后续可视化和数据库存储。

这样做的目的是让模型差异主要来自检测范式本身，而不是数据组织、脚本实现和结果格式的差异。如果不同模型各用各的样本划分、各用各的评测脚本、各用各的结果格式，就算得到几个看起来可比的数字，也很难据此形成可靠结论。所以本文把"统一格式"当作实验方法的一部分，而不只是训练前的工程准备。

统一格式还带来了两个直接好处。一是 MMDetection、YOLO 和本地插件推理结果可以通过同一套后处理与可视化逻辑整理，降低了跨框架联调成本。二是结构化结果能直接复用进 Web 系统数据库，让实验章的离线结果和系统章的在线任务回放使用同一套数据表达。这个设计保证了论文实验链路和系统实现链路之间的自然衔接。

\subsection{评价指标}

本文用 AP50 作为主指标，AP50-95、Precision、Recall 和 F1 作为辅助指标。相关定义如下。

\begin{equation}
\text{Precision} = \frac{TP}{TP + FP}
\end{equation}

\begin{equation}
\text{Recall} = \frac{TP}{TP + FN}
\end{equation}

\begin{equation}
F1 = \frac{2 \cdot \text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}
\end{equation}

\begin{equation}
\text{IoU} = \frac{|B_{pred} \cap B_{gt}|}{|B_{pred} \cup B_{gt}|}
\end{equation}

AP50 是 IoU 阈值取 0.5 时的平均精度，反映模型在当前任务中的整体检测能力。AP50-95 在多个 IoU 阈值下做平均，更强调定位质量和框回归能力。Precision 和 Recall 分别反映误检控制和漏检控制能力，F1 综合刻画二者的平衡。除了这些精度指标，本文还在后续实验中统计了 FPS、参数量和 FLOPs，用来辅助分析模型的工程部署属性。

\subsection{指标选择依据}

选 AP50 作为主指标，是因为微小船舶目标在图像中尺寸太小了——预测框和标注框差几个像素，IoU 就会受很大影响。如果用更严格的高 IoU 阈值来评价，容易放大框回归误差而弱化"是否检出目标"这个核心问题。对遥感微小目标检测来说，AP50 更适合先回答"模型能不能稳定发现目标"，AP50-95 则用来补充说明"检出的框够不够准"。

Precision、Recall 和 F1 在本文中有同等重要的解释价值。DRENet、FCOS 和 YOLO26 的设计目标本来就不完全一样，单看 AP50 看不出它们对误检和漏检的不同取舍。高 Recall 的模型更适合强调漏检控制的场景，高 Precision 的模型更适合强调输出稳定性和工程部署可控性的场景。所以本文在后续实验分析中始终把 Precision--Recall 平衡作为结论解释的重要依据，而不是只用单一分数做简单排序。

\subsection{模型方案}

为了体现不同检测范式在遥感微小船舶任务中的表现差异，本文选了三类代表模型做对比实验。

\subsubsection{DRENet（专项方法）}

DRENet 是针对遥感微小船舶任务提出的专项方法，通过退化重建增强机制在训练阶段强化对弱纹理目标的表示能力\cite{chen2022_tinyship_drenet}。该方法对复杂海面背景有较强针对性，是本文的重要主线模型之一。本文重点关注它在 Recall 和复杂背景场景中的表现。

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.95\textwidth]{img/ch3_drenet_overview.png}
  \caption{DRENet 方法示意}
  \label{fig:ch3_drenet_overview}
\end{figure}

如图~\ref{fig:ch3_drenet_overview} 所示，DRENet 在常规检测路径之外额外引入了退化重建增强分支。这个分支只在训练阶段参与，通过重建带有目标感知特性的退化图像来约束骨干特征学习，让网络更倾向于关注微小船舶而不是大片背景纹理。CRMA 模块进一步增强了跨阶段特征的判别能力。因此该方法在本文实验中更容易表现出"召回优先"的检测特征。

\subsubsection{FCOS（Anchor-Free 参考基线）}

FCOS 通过位置回归替代锚框设计，降低了 anchor 尺寸和比例等超参数依赖\cite{tian2019_fcos}。在本文中，FCOS 的角色更接近 anchor-free 参考基线——它用来帮助分析"在不依赖专项结构的情况下，通用 anchor-free 方法在遥感微小船舶任务中能达到怎样的效果"。需要说明的是，当前可用的 FCOS 正式结果采用 MMDetection 默认输入策略，因此它主要承担参考和补充比较作用，不作为严格同输入尺度公平主结论的核心对象。

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.95\textwidth]{img/ch3_fcos_pipeline.png}
  \caption{FCOS 检测流程示意}
  \label{fig:ch3_fcos_pipeline}
\end{figure}

图~\ref{fig:ch3_fcos_pipeline} 展示了 FCOS 的核心思路：输入图像经过 Backbone 与 FPN 提取多层特征后，不再依赖预设 anchor，由检测头直接预测类别、边框偏移和 centerness。这种设计降低了锚框超参数带来的额外调节成本，也使 FCOS 成为本文中最具代表性的 anchor-free 参考路线。正因为结构更通用，FCOS 在论文中更适合承担范式对照而不是专项最优模型的角色。

\subsubsection{YOLO26（One-Stage 轻量部署代表）}

YOLO 系列以端到端检测和较高推理效率著称，是工程部署中应用广泛的一类模型\cite{redmon2016_yolo,redmon2018_yolov3,ultralytics2024_docs}。本文采用 YOLO26 作为 one-stage 路线代表，重点考察它在精度、定位质量、训练链路成熟度和系统接入便利性之间的平衡关系。

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.95\textwidth]{img/ch3_yolo_pipeline.png}
  \caption{YOLO26 单阶段检测流程示意}
  \label{fig:ch3_yolo_pipeline}
\end{figure}

从图~\ref{fig:ch3_yolo_pipeline} 可以看到，YOLO26 延续了典型 one-stage 路线的结构——Backbone 提取特征、Neck 完成多尺度融合、检测头一次性输出候选框结果。相比需要更强专项设计的模型，这种结构在训练、部署和接口封装上都更直接，因此特别适合作为本文系统默认模型候选。它在实验中的优势也主要体现在整体精度、定位质量与接入便利性的综合平衡上。

\subsection{统一实验协议与实现约束}

为了保证实验结论可比，本文对实验协议做了统一约束：使用同一数据划分和同一评测口径；DRENet 和 YOLO26 统一采用 512 输入主线，FCOS 在论文中明确为历史正式 run 对应的参考基线并对输入尺度差异进行说明；统一记录实验环境、配置文件、权重路径、日志文件和关键结果；统一结果输出结构，使算法结果能直接复用到后续系统实现中。

这个统一协议并不意味着所有模型在每个细节上都完全同构。本文明确保留了模型实现差异的客观边界，并把这些边界写进了实验说明。比如 FCOS 当前可用的正式结果来自 MMDetection 默认输入策略，在论文中只作为 anchor-free 参考基线，不参与严格同输入尺度的主结论竞争；DRENet 在当前本地插件路径下对非 512 输入存在 shape mismatch，在消融实验中明确写为实现边界，而不是强行生成不可靠结果。这种处理方式虽然更保守，但能保证结果说明与实际实现保持一致。

在实现层面，本文通过统一推理入口完成不同模型的封装：

\begin{verbatim}
predict(image_path, model_name, conf_threshold, iou_threshold)
\end{verbatim}

在此基础上又进一步封装了融合推理接口和系统任务执行逻辑。这样做的意义在于把算法层和系统层解耦，让模型训练、离线评测、单图推理、批量推理和 Web 系统接入共享同一套核心能力，不需要为不同入口重复编写逻辑。

统一推理入口不只是代码复用问题，也是实验方法问题。如果离线评测、命令行可视化和系统任务接口各自维护各自的参数语义和结果结构，论文结果、演示结果和数据库记录之间就会出现口径偏移。采用统一入口后，阈值设置、模型选择、结果字段和可视化流程在不同场景下保持一致，把"实验结果可信"和"系统展示可信"统一到了同一实现基础上。

\subsection{小结}

本章介绍了本文实验所依赖的数据集、统一预处理流程、评价指标和三类模型方案，并说明了统一实验协议与实现约束。通过这些设计，本文建立了面向遥感微小船舶检测任务的可复验实验基础，也为后续的实验结果分析和系统实现提供了统一接口与数据支撑。
