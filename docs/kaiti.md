### 2. 具体怎么做（按顺序做，做完就能写开题）

#### 2.1 先把“问题”说清楚（1 页内能讲明白）
你的题目是“遥感场景下的微小船舶检测”。开题里建议用下面 5 句话把问题定义写死：
1) **输入**：遥感图像（卫星/航空），分辨率高、背景复杂（海面/港口/波纹/浪花/云雾）。  
2) **目标**：图像中船舶目标往往尺度很小（tiny/small），易漏检。  
3) **输出**：船舶的目标框（bbox）及置信度（可扩展为批量推理与结果导出）。  
4) **挑战**：小目标特征弱、多尺度变化大、背景干扰强、类别/目标稀疏、成像退化。  
5) **意义**：海上监管/航运管理/安全巡检/应急救援等。

开题里要体现“你能解决什么”：不是承诺 SOTA，而是承诺**可复现的强基线 + 对比分析 + 可用系统原型**。

#### 2.2 文献调研（20–30 篇）怎么做，才能写出“研究现状”
老师给了两条高效路线，你可以直接照做：

**路线 1：先找 Survey（综述）再反向扩展**
- 关键词：`remote sensing object detection survey`、`small object detection survey`、`tiny object detection`、`ship detection remote sensing survey`
- 做法：选 1–2 篇近年的 survey，按它的分类体系抄一份你的“分类框架”，然后从其引用中挑与你题目最相关的 15–20 篇。

**路线 2：找近两年 SOTA paper，从实验对比与 related work 反向扩展**
- 关键词：`tiny ship detection remote sensing`、`LEVIR-Ship`、`TGRS tiny ship`、`small object detection transformer`
- 做法：挑 3–5 篇近两年 paper，重点看：
  - `comparison with state-of-the-art`（对比了哪些方法）
  - `related work`（提到哪些方向）
  - 你把出现频率高、被反复对比的论文补齐到 20–30 篇

**建议你把 20–30 篇分成 4 类**（写研究现状会非常顺）：
- **通用目标检测**：Faster R-CNN/RetinaNet/FCOS/DETR 及其改进
- **小目标检测专门方法**：多尺度、注意力、超分/增强、蒸馏、特征融合等
- **遥感/船舶检测专门方法**：遥感数据集、海面背景、小目标分布、旋转框（如涉及）
- **工程与工具链**：mmdetection/ultralytics 的复现经验（可作为“工程实现现状”补充）

**最实用的阅读笔记模板（每篇 5 分钟能填）**
- 论文一句话要解决什么问题？
- 用了什么关键方法（模块/损失/结构）？
- 用了什么数据集（是否含 LEVIR-Ship）？
- 用了什么指标（mAP/AP50/Recall 等）？
- 结论是什么、你能借鉴什么？

**开题里“研究现状”怎么写（建议 3–5 页）**
- 先给分类框架（1 段 + 1 张小表/小图）
- 每一类挑 4–8 篇代表作，按时间/方法演进写
- 最后一段写“不足与机会”（为你的工作铺垫）：
  - 现有方法在微小船舶上仍存在漏检/误检
  - 很多论文缺少统一对比、或工程落地难
  - 因此你将做：统一数据与指标、复现强基线、系统集成与可视化

#### 2.3 数据集与评价指标（开题必须把“实验怎么评”说清楚）
你的题目老师指定了数据集与参考实现：
- 数据集：LEVIR-Ship（`https://github.com/WindVChen/LEVIR-Ship`）
- 参考方法代码：DRENet（`https://github.com/WindVChen/DRENet`）

开题里你需要写清楚 4 件事：
1) **数据来源与规模**：训练/验证/测试各多少张（若你还没下载到，可先写“以官方划分为准，后续补充统计”）
2) **标注形式**：bbox（水平框）还是旋转框（以数据集实际为准）

#### 2.5 系统要怎么做（开题阶段写“需求 + 设计”，不用把代码都写完）
开题里系统章节（第 5 部分）建议最少覆盖：

**需求分析（Functional）**
- 支持选择模型（≥3：DRENet + mmdet + YOLO）
- 支持单张/批量推理
- 输出可视化结果图（带检测框/置信度）
- 导出结构化结果（JSON/CSV）

**非功能需求（Non-functional）**
- 可复现安装（conda/requirements）
- GPU 优先，CPU 兼容（可选）
- 推理速度要求（给个目标即可，如“单张 < 1s@GPU”）

**总体设计**
- 统一推理接口：`predict(image, model_name, conf, iou)`
- 模块划分：数据预处理 / 模型适配层 / 推理与后处理 / 可视化导出 / UI（CLI 或 Web）
- UI 选择建议：为了答辩演示省事，优先 **Gradio** 或 **Streamlit**（开题可先写方案）

#### 2.7 风险与备选方案（开题写了会很加分）
常见风险与对策（直接写进开题即可）：
- **算力不足/训练太慢**：降低输入尺寸、先用轻量模型（YOLO-n/s）、减少 epoch 做预实验
- **数据格式适配困难**：先统一转 COCO 格式，再分别适配 mmdet/YOLO
- **复现参考代码失败**：以 mmdet/YOLO 作为可交付主线，同时保留 DRENet 复现作为增强；或使用论文提供的预训练权重（如有）
- **指标提升不明显**：强调“对比与分析 + 系统实现”为核心成果，不把“提升 SOTA”作为唯一目标
