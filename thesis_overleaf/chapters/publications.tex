\chapter{系统需求分析、设计与实现}
\label{ch:system}

\section{系统建设目标}

本文的系统建设目标不是搭建面向生产环境的复杂平台，而是完成一套面向本科毕业设计场景、能够真实运行、稳定演示并支撑论文结果展示的轻量级检测系统。该系统既要体现前端、后端、数据库和算法集成的完整性，又要控制实现复杂度，避免因引入过重的分布式组件而影响毕业设计主线推进。

基于这一目标，本文最终采用 React + FastAPI + SQLite 的本地单机方案。选择该方案的原因主要有三点：第一，React 能够较好地支撑任务提交、列表展示和结果详情页等交互界面；第二，FastAPI 便于快速构建结构清晰的 HTTP 接口，并与 Python 推理代码自然衔接；第三，SQLite 足以满足毕业设计阶段对任务持久化和结果回看的需求，同时不需要额外部署独立数据库服务。通过这样的技术组合，系统既具备完整三层结构，又保持了实现上的轻量与可控。

\section{系统需求分析}

\subsection{功能需求}

结合毕业设计任务要求和答辩演示需求，本文系统的核心功能可以概括为以下几个方面。

\begin{enumerate}
  \item \textbf{模型管理}：能够读取模型配置，展示模型名称、状态和权重信息，并支持启用或停用模型；
  \item \textbf{任务提交}：能够支持单张图像和批量图像上传，并允许用户设置推理模式、模型类型和置信度阈值；
  \item \textbf{任务执行}：能够以异步方式执行推理任务，避免前端界面阻塞，并支持任务状态查询；
  \item \textbf{结果展示}：能够展示检测框可视化图像、结构化检测结果表格和统计信息；
  \item \textbf{结果持久化}：能够在系统重启后继续保留历史任务、结果和输出文件路径，支持回放与复查。
\end{enumerate}

\subsection{非功能需求}

除功能需求外，系统还应满足一定的非功能要求。首先，系统应具备良好的\textbf{可复现性}，即环境安装、数据库初始化和项目启动方式清晰可重现。其次，系统应具备\textbf{可维护性}，即算法层、服务层和展示层职责边界明确，后续更换模型或增加新接口时不需要推翻已有结构。再次，系统应具备\textbf{可演示性}，界面信息需要简洁、清楚并适合答辩展示。最后，系统应坚持\textbf{轻量化}原则，不引入 Redis、Celery 或分布式部署等超出本科毕业设计需要的复杂组件。

\section{系统总体架构}

系统总体采用前后端分离架构，如图~\ref{fig:ch5_architecture} 所示。前端负责任务创建、列表查询和结果可视化；后端负责接口服务、任务调度、状态管理与静态文件访问；数据库负责持久化模型、任务、结果和输出文件路径；算法能力则由已有推理内核提供，并通过统一接口被后端复用。

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.96\textwidth]{generated/ch5_architecture.png}
  \caption{系统总体架构示意}
  \label{fig:ch5_architecture}
\end{figure}

从职责划分看，系统可分为以下五层。

\begin{enumerate}
  \item \textbf{前端展示层}：基于 React、Vite、TypeScript 和 Ant Design 实现任务提交页、任务列表页、任务详情页和模型管理页；
  \item \textbf{后端服务层}：基于 FastAPI 提供任务提交、任务查询、结果查询、模型查询和健康检查接口；
  \item \textbf{任务调度层}：基于 \texttt{ThreadPoolExecutor} 实现本地异步任务执行，避免显存冲突和界面阻塞；
  \item \textbf{数据持久化层}：基于 SQLite 与 SQLAlchemy 保存模型信息、任务状态、检测结果和文件路径；
  \item \textbf{算法能力层}：复用现有 \texttt{src/} 目录中的预测服务、模型适配器和融合逻辑，实现单模型与融合模式推理。
\end{enumerate}

如图~\ref{fig:ch5_architecture} 所示，系统把页面交互、接口服务、任务调度、算法运行和结果存储拆分为相对独立的层次。其中，前端仅通过 HTTP 接口与后端交互，后端通过统一推理入口复用已有算法内核，数据库与输出目录共同承担任务状态和产物持久化职责。这样的分层方式使系统既能在答辩场景下稳定运行，也为后续替换模型或扩展功能保留了清晰接口。

\section{数据库设计}

为了满足任务追踪、结果回看和产物管理需求，系统设计了四张核心数据表，即 \texttt{models}、\texttt{tasks}、\texttt{results} 和 \texttt{task\_files}。其中，\texttt{models} 表用于管理模型名称、键值、权重路径和启停状态；\texttt{tasks} 表用于记录任务类型、执行模式、状态流转和时间信息；\texttt{results} 表用于保存结构化检测框结果；\texttt{task\_files} 表用于记录输入图、输出 JSON 和可视化结果路径。

  \caption{系统数据库设计示意}
  \label{fig:ch5_er}

结合图~\ref{fig:ch5_er} 可以看到，\texttt{tasks} 表处于系统数据链路的中心位置，一方面与 \texttt{results} 表形成任务到检测框结果的一对多关系，另一方面与 \texttt{task\_files} 表形成任务到产物文件的一对多关系。\texttt{models} 表则承担模型配置与启停状态记录功能，为前端模型管理页和后端推理校验提供统一来源。上述设计体现了“最小但完整”的思路：它已经覆盖模型管理、任务调度、结果回放和产物索引的完整链路，同时又没有引入用户、权限、多租户等超出本科毕设范围的扩展实体。

\section{关键实现流程}

\subsection{任务创建与执行流程}

当用户在前端页面上传图像并选择任务参数后，前端通过 \path{/api/v1/tasks/infer} 接口将请求发送到后端。后端首先对图像、模式、阈值和模型配置进行校验，然后创建任务记录并写入数据库。随后，任务被提交到本地线程池执行器，由执行器根据参数选择单模型或融合模式推理，并调用统一推理接口完成实际检测。

在任务执行过程中，系统会对状态进行显式管理，包括 \texttt{queued}、\texttt{running}、\texttt{done} 和 \texttt{failed} 四类状态。这样做的好处是，前端列表页和详情页能够统一展示任务生命周期，也便于系统在异常情况下给出明确反馈。

\subsection{结果生成与展示流程}

推理完成后，系统会把结构化检测结果写入数据库，并将输入图、输出 JSON、可视化图像等产物统一保存到 \texttt{outputs/tasks/<task\_id>/} 目录下。前端通过轮询任务详情接口和结果接口，展示任务进度、可视化图像、融合结果和检测表格。这样，用户不仅可以看到最终是否检测到目标，还可以查看具体框信息、模型来源和历史任务记录。

\section{系统实现效果与工程取舍}

经过前后端联调和真实模型接入验证，系统已经完成从任务创建到结果回放的完整链路。当前系统具备以下实现效果：能够接入 DRENet 与 YOLO 的真实权重并完成单图与批量推理；能够支持任务列表轮询刷新、进度条展示和失败状态标记；能够在详情页展示模型来源、检测框、平均置信度和任务统计图表；能够使用 SQLite 保留历史任务和结果，保证系统重启后记录仍可查询。

\subsection{前端页面实现效果}

在页面层面，本文围绕“简洁、清楚、适合答辩展示”的目标，对任务提交页、任务列表页、任务详情页和模型管理页进行了统一设计。任务提交页支持单图与批量两类输入方式，并将推理模式、模型选择和阈值参数集中在同一表单区域，降低了用户在演示过程中的操作路径。任务列表页重点突出任务状态、进度、模式、创建时间和耗时信息，便于快速浏览系统当前运行情况。任务详情页则进一步展示可视化结果、结构化检测框、模型来源和统计图表，使单次任务的执行过程和结果信息能够被完整回看。

与传统“仅展示一张结果图”的桌面可视化方式相比，Web 前端的优势在于，它把任务参数、执行状态、结果图像和结构化数据纳入同一交互空间。这样一来，论文中的实验结论不仅可以通过表格和段落说明，也可以在系统中通过任务记录得到直观展示。这种展示方式更符合本科毕业设计答辩的实际场景，即教师既关心模型性能，也关心系统是否真正可用、结果是否能够回放和核对。

从工程取舍角度看，本文没有采用更重的企业级方案，例如消息队列、分布式任务系统或独立数据库服务器，而是选择本地单机方案。这一选择并不是功能退让，而是基于毕业设计目标做出的合理折中：其一，单机方案已经能够完整体现前端、后端、数据库和算法集成能力；其二，轻量方案更有利于在有限时间内保证系统稳定运行；其三，系统复杂度降低后，可以把更多精力投入到实验结果和论文论证本身。

这一工程取舍还体现在任务调度方式上。本文没有引入 Redis、Celery 等额外中间件，而是采用基于 \texttt{ThreadPoolExecutor} 的本地异步执行器，默认串行调度推理任务。对于真实生产环境而言，这种方式在扩展性上显然不如分布式队列方案，但对于当前毕设场景，它能够有效规避显存冲突、部署复杂度和跨服务联调成本，同时又保留了任务化执行、状态管理和失败回写等核心系统特性。因此，这种设计并不是“简单起见”的权宜之计，而是围绕目标场景做出的有意识简化。

此外，系统对 SQLite 的使用也体现了相同思路。SQLite 并不是为了模仿企业级数据库方案，而是因为本文的核心诉求是完成任务记录、结果回看和输出文件索引的持久化。对于单机本地演示和答辩场景而言，SQLite 已经能够稳定支撑这类需求，并避免了独立数据库服务带来的额外部署负担。换言之，本文系统追求的是“最小但完整”的三层闭环，而不是为了复杂而复杂。

因此，本文系统的价值不在于追求大而全，而在于以适度复杂度完成一个真实可运行、结构清晰、结果可追溯的检测平台。这种“轻量但完整”的实现路径更符合本科毕业设计的目标定位。
